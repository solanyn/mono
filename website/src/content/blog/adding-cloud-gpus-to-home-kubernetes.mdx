I run a Kubernetes cluster at home with three low-power Talos nodes, enough for most things, but when I want to train models or run GPU workloads, I don't have a good solution. I'm not interested in dropping money on a dedicated GPU for subpar performance.

What if we could just build something to burst to cloud GPUs when we need to?

## Sidequests

### Talos GPU Provider Operator

My first attempt was [`tgp-operator`](https://github.com/solanyn/tgp-operator), what if we could just create a control plane for cloud providers? It went great! At first. Then we ran into problems. Provider API fragmentation meant lots of glue code and many of them didn't have API docs. I didn't want to front cash for capacity I might not use, some providers want you to load up a minimum number of credits before they'll let you even provision a GPU node. The biggest problem I had was, if my cluster is in a private network, how are we gonna make secure connections from the cloud to my cluster at home? I'm not putting my cluster endpoint publicly (neither would most on-prem cluster setups). Wireguard and Tailscale are awesome, but I really didn't want to build in networking fabric _just_ to use a cloud node. In the bin it goes.

### Modal

I've been keeping an eye on [Modal](https://modal.com/) since they launched, their infrastructure as SDK model is really cool. For my case, it's a bit too limiting, I can't schedule any pod I want and keeping everything in the Kubernetes ecosystem for monitoring reasons, turns out was pretty important to me. I built [modal-operator](https://github.com/solanyn/modal-operator) with a CRD for a `ModalApp` that gets the Modal URL and slaps it in a Kubernetes service. But that only works for things like LLM endpoints and maybe other scale-to-zero API services, no model training workloads unless we go all in on doing things the Modal way.

Kinda neat, but didn't entirely fit the bill.

## Liqo

I found Liqo when looking at CNCF projects. Maybe I needed cloud federation. Liqo peers clusters together to make them _relatively_ seamlessly. Kubernetes will happily schedule onto Liqo nodes in a second cluster as "virtual nodes".

  VN -.->|Liqo Tunnel| GPU2

Paired it with GKE using spot instances:

- Home cluster: always on, control plane, lightweight workloads
- GKE: small base node pool for Liqo pods, GPU node pool with spot T4s + general compute nodes, scales to zero when idle

### Offloading

When I submit a GPU workload, Liqo:

<Mermaid
  code={`sequenceDiagram
  participant User
  participant Home as Home API Server
  participant Liqo as Liqo Controller
  participant VN as Virtual Node
  participant GKE as GKE Cluster
  
  User->>Home: kubectl apply (GPU pod)
  Home->>Liqo: Pod needs nvidia.com/gpu
  Liqo->>VN: Schedule to virtual node
  VN->>GKE: Create pod via peering
  GKE->>GKE: Scale up T4 spot node
  GKE-->>VN: Pod running
  VN-->>Home: Status update
  Home-->>User: Pod ready
`}
/>

The pod runs on GKE hardware but appears in my home cluster and monitoring, ingress/egress work transparently.

### $$$

| Resource       | Type           | $/hr  | Max nodes | Max $/mo | Notes                          |
| -------------- | -------------- | ----- | --------- | -------- | ------------------------------ |
| GKE management | —              | $0.10 | 1         | $0       | free tier covers first cluster |
| Liqo base      | e2-small spot  | $0.01 | 2         | ~$15     | always on                      |
| GPU            | T4 spot        | $0.35 | 8         | ~$2,044  | scales to 0                    |
| Compute        | e2-medium spot | $0.01 | 4         | ~$29     | scales to 0                    |

Theoretical max: ~$2,088/mo if everything ran 24/7. Node pools scale to zero when idle, I only pay for what I use. Something to note is that the node counts are easily configurable since the GKE cluster is GitOps managed with [crossplane](https://github.com/crossplane/crossplane) and the caps are by design so we can't scale to infinity (it's not supposed to). Adding or extending node pools is just one manifest away.
